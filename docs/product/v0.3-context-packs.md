# VibeVoice 0.3: Context Packs

**Status:** Approved product direction
**Target version:** `0.3.0`
**Validated against:** VibeVoice `0.2.6` on `master`
**Date:** July 17, 2026

## Decision

VibeVoice will become an opinionated, fully local voice-to-prompt compiler for developers.

```text
Hold hotkey
-> speak naturally
-> apply an explicit developer workflow
-> compile speech with local vocabulary and deterministic rules
-> safely insert an inspectable result
```

The product will not compete by accumulating transcription engines, meeting features, mobile clients, cloud accounts, or general-purpose assistant behavior. It will deepen the existing local workflow around developer-intent fidelity, bounded context, predictable processing, and safe insertion.

## Product position

> VibeVoice is a local voice-to-prompt compiler for developers that turns natural speech into precise input for coding agents, editors, and technical communication.

Across the `0.3.x` line, explicit project metadata and terminal policies extend that compiler to project-aware prompts and safer terminal input.

Context Packs are the organizing product concept. The concept is not claimed as a novel configuration primitive; competitors already provide modes and profiles. VibeVoice's differentiation is the combined contract:

- Local operation on Linux, Windows, and macOS
- Deterministic processing before any optional probabilistic stage
- Metadata-only project vocabulary with no default source-code ingestion
- Inspectable transformation and context receipts
- Plain-text clipboard preservation and stale-target protection
- Explicit terminal-safe behavior
- No mandatory account, cloud service, or subscription

## Product principles

1. **Local by default.** Recording, transcription, processing, context collection, receipts, and history remain on the workstation.
2. **Deterministic first.** Every built-in `0.3` transformation must be explainable and repeatable.
3. **Explicit context.** Context sources are named, individually permissioned, visible during use, and recorded only when policy allows.
4. **Safe output.** Insertion is a planned operation with a target and policy, not an unconditional paste side effect.
5. **Developer depth over category breadth.** VibeVoice improves agent prompts, technical chat, commits, and terminals without becoming a meeting or notes platform.
6. **Existing behavior remains available.** Upgraded users retain a Raw Dictation pack compatible with the current workflow.

## Release scope

### P0: foundations and blockers

- Resolve renderer issue [#15](https://github.com/Zburgers/vibevoice/issues/15)
- Resolve CSP and capability isolation issue [#33](https://github.com/Zburgers/vibevoice/issues/33)
- Preserve previous clipboard text safely
- Detect target changes before insertion
- Define terminal-safe insertion policy
- Extract the Rust backend into behavior-preserving modules

P0 may ship independently as `0.2.7`. The Context Pack foundation remains a `0.3.0` capability.

### P1: Context Pack foundation

- Versioned Context Pack storage and migrations
- Built-in Raw Dictation, Agent Prompt, and Developer Chat packs
- Manual pack selection and optional pack-specific hotkeys
- Structured transcript document and deterministic processor stages
- Spoken formatting commands and paragraph preservation
- Transformation receipts
- Copyable diagnostics report from [#8](https://github.com/Zburgers/vibevoice/issues/8)

### P2: developer context

- Active-application identity and activation rules
- Terminal Safe Mode
- Commit and Pull Request pack
- Metadata-only project vocabulary
- Per-application context exclusions
- Context indicators and receipts
- Per-application activation and target policies

### P3: quality and engine flexibility

- Internal transcription-engine contract with `WhisperCliEngine` as the only first implementation
- Local corpus and hardware benchmark
- Model discovery and measured recommendations for [#11](https://github.com/Zburgers/vibevoice/issues/11)
- History reprocessing and searchable processing metadata when justified

### P4: optional local intelligence

An optional local language model may perform predefined transformations only after deterministic Context Packs are proven. It must be disabled by default, explicitly labeled probabilistic, inspectable, and time-bounded. Model-introduced commands, files, tests, or implementation claims must never be presented silently as user-provided; material changes require a visible preview and receipt, with a deterministic fallback.

## Built-in Context Packs

| Pack | Initial phase | Purpose | Default insertion policy |
| --- | --- | --- | --- |
| Raw Dictation | P1 | Compatible, minimal cleanup for general text | Auto-paste where supported |
| Agent Prompt | P1 | Structured Markdown prompts for coding agents | Auto-paste with target-change guard |
| Developer Chat | P1 | Conversational technical communication | Auto-paste with target-change guard |
| Terminal Safe Mode | P2 | Exact shell and REPL tokens | Preview; never press Enter |
| Commit and Pull Request | P2 | Titles, bodies, lists, and issue references | Preview or copy by default |

Post-processing presets in [#10](https://github.com/Zburgers/vibevoice/issues/10) and recording profiles in [#12](https://github.com/Zburgers/vibevoice/issues/12) are one product layer under Context Packs. The documentation does not close or rewrite either issue; issue disposition belongs to implementation planning.

## Explicit non-goals for 0.3

- Meeting recording or speaker diarization
- Mobile companion or account system
- Cloud history, team workspaces, or enterprise administration
- Prompt or plugin marketplace
- Multiple hosted transcription backends
- Generic desktop automation or automatic command execution
- Permanent audio retention
- Arbitrary source-code indexing
- Default LLM rewriting

## `0.3.0` success criteria

### Reliability

- At least 99% successful insertion across the declared supported application matrix
- No unintentional terminal command execution
- No stale-target auto-paste in automated scenarios
- Plain-text clipboard restoration never overwrites a newer user clipboard value
- No lost or duplicated history entries in concurrency tests

### Responsiveness

- Hotkey to visible `Preparing`: under 100 ms
- Hotkey to active recording: under 300 ms on supported hardware
- Pill state update: under 200 ms
- No renderer disappearance across 500 state cycles
- No full history reload for meter events or runtime-only state updates

Transcription latency will be benchmarked by hardware class rather than assigned one universal threshold.

### Output quality

- At least 30% fewer incorrect developer-vocabulary terms than `0.2.6` on the maintained P1 corpus
- At least 95% correct spoken-formatting command recognition
- No dictionary substring corruption
- Paragraph and Markdown structure preserved

### Privacy

- No network dependency in the default workflow
- No context capture without a visible policy
- Complete local context and transformation receipts
- One action to clear retained history, receipts, and local quality metrics

## P2 success criteria

- At least 30% fewer incorrect project-specific terms than the P1 baseline
- Terminal tokens preserved exactly in the terminal corpus
- No project-vocabulary source-code scanning
- One action to clear project vocabulary and context caches
- Per-application activation and exclusion policies pass the compatibility matrix

## Release gate

`0.3.0` is ready only when P0 and P1 acceptance criteria pass on the supported platform and application matrix. P2 may follow in a later `0.3.x` release if it cannot meet the same safety bar without delaying a coherent P1 release.
